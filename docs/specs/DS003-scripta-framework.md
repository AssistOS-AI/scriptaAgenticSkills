# DS003 SCRIPTA Framework

## Introduction

This file defines the project-level architecture of SCRIPTA. It translates STG into a staged system for literary production and establishes the boundaries between planning, generation, validation, and editorial finishing.

## Core Content

SCRIPTA must be grounded in STG as its theoretical source. The framework must preserve the idea that narrative writing is a structured process rather than an unbounded act of inspiration. The system therefore has to expose explicit stages, explicit artifacts, and explicit handoffs between those stages.

The public subsystem sequence must be:

| Stage | Primary role | Dominant execution style | Main output |
| --- | --- | --- | --- |
| MacroPlan | Define the book skeleton | Skill-orchestrated, seed-first | Macro package |
| ChapPlan | Convert the skeleton into chapter progression | Skill-orchestrated, seed-first | Chapter plan set |
| MicroPlan | Convert chapters into symbolic block plans | Script-first | Symbolic micro plans |
| CNLEnh | Add constrained natural-language hints and resolve placeholders | Skill-mediated LLM refinement | Enhanced CNL plans |
| ChapGen | Draft chapters from plans | Skill-mediated LLM drafting | Chapter drafts |
| BookWriter | Perform editorial and translation finishing | Skill-mediated LLM drafting | Export-ready manuscripts |
| Validation Suite | Evaluate narrative and compliance quality | Hybrid, script-first | Reports and raw evidence |
| Metrics Framework | Normalize and aggregate scores | Script-first | Percent indicators |

All planning subsystems must communicate through a shared CNL plan language. MacroPlan, ChapPlan, and MicroPlan must emit initial command blocks in that language. CNLEnh must enrich those same plans through later CNL refinements rather than by replacing them with opaque prose notes. The canonical CNL contract is defined in `DS013-cnl-plan-format.md`, the command set is defined in `DS014` through `DS028`, and the shared symbolic seeding layer is defined in `DS029-symbolic-plan-generation.md`.

The framework must preserve a one-directional dependency principle. A later stage may read and reinterpret earlier artifacts, but it must not silently redefine the upstream contract. If a later stage discovers a contradiction that cannot be resolved safely, it must emit a warning or send the workflow back for revision rather than overwriting upstream intent in place.

The planning pipeline must begin with symbolic seed generation. Skills may orchestrate that seed generation through CLI tools, but the first plan iteration must always exist as a concrete artifact before an LLM is asked to refine it. The framework must therefore treat symbolic generation as a foundational planning capability, not as an optional implementation detail.

The framework must also preserve stage immutability. A stage may consume prior artifacts and create successor artifacts, but it must not destroy or mutate the earlier stage outputs that validators and reviewers need for comparison.

SCRIPTA must preserve traceability at every stage. Every artifact must retain stable book, chapter, and local unit identifiers so that generated text, validation findings, and aggregated metrics can all point back to the same structural lineage.

The framework must distinguish public project contracts from repository helper tooling. Internal helper skills may be used to author documents or bootstrap the repository, but they must not become public SCRIPTA subsystems in `docs/` or in the DS set.

The current reference runtime is implemented at the repository root through `bin/scripta.mjs`, `runtime/`, and the SCRIPTA skill wrappers under `.agents/skills/`. It also ships with a generated `QA/` bundle that exercises the five baseline profiles end to end.

## Decisions & Questions

### Question #1: Why does the framework insist on explicit handoffs instead of letting later stages re-derive missing information?

Response: Re-derivation hides errors and makes validation difficult. The project needs explicit handoffs so that quality issues can be traced to specific planning or generation decisions rather than being diffused across opaque regeneration steps.

### Question #2: Why is validation modeled as a subsystem instead of a final optional review?

Response: The user asked for scripts, metrics, reports, and issue localization as core outputs. Validation is therefore a built-in stage of the architecture, not a discretionary afterthought.

### Question #3: Why is symbolic plan generation called out explicitly if MacroPlan, ChapPlan, and MicroPlan already exist?

Response: The user clarified that the first iteration of plan files should be generated by code through parameterized tooling and only later improved by LLM-mediated skills. The framework therefore has to reserve explicit room for symbolic seeding rather than describing the planning stack as LLM-originated.

## Conclusion

SCRIPTA must remain a staged, traceable, STG-grounded architecture. Future implementation work must preserve these subsystem boundaries and the handoff discipline between them.
